# Contributing Policy

Thank you for your interest in Sovereign System Architecture. This repository is maintained as a controlled Public Edition and flagship authority hub for the ecosystem.

## Contribution Model

External code contributions are not accepted unless explicitly approved by the repository owner in advance.

This restriction protects the Public Edition and prevents accidental disclosure of private enterprise implementation details, commercial materials, client-specific architecture, security-sensitive controls, compliance evidence, prompts, or operational methods.

## Contributions That May Be Considered

The maintainer may selectively consider non-sensitive contributions such as:

- typo fixes or documentation clarity improvements
- governance file improvements
- accessibility notes for public documentation
- issue reports about broken links or inaccurate public claims
- safe suggestions for public repository hygiene

Please open an issue before preparing any pull request. Unapproved pull requests may be closed without review.

## Contributions That Will Not Be Accepted

Do not submit, request, or describe:

- private client architectures, customer environments, infrastructure maps, or internal network diagrams
- production deployment methods, hardening playbooks, access-control policies, or trust-boundary rules
- private AI prompts, system prompts, tool instructions, model configuration, or evaluation traces
- API keys, tokens, service-account files, credentials, environment variables, or secrets
- private compliance evidence, audit artifacts, DORA/NIS2/GDPR records, or regulator-facing materials
- private integrations with cloud accounts, identity providers, internal databases, SIEM/SOC tools, or enterprise systems
- incident-response procedures, operational runbooks, escalation paths, or security-control implementations
- commercial implementation details, acquisition materials, pricing models, or buyer-specific requirements
- changes that enable unauthorized access, data misuse, credential misuse, unsafe deployment, compliance misrepresentation, or extraction of private implementation methods

## Public Edition Boundary

The Public Edition communicates safe architecture direction, flagship ecosystem authority, governance maturity, brand positioning, and commercialization readiness. It is not the Private Enterprise Edition and does not accept contributions that attempt to recreate private implementation details.

## Pull Request Expectations

If the maintainer explicitly invites a contribution:

1. Keep the change narrow and documentation-focused unless otherwise approved.
2. Do not modify design, visuals, assets, architecture content, branding, README positioning, or private implementation details unless specifically requested by the maintainer.
3. Avoid adding dependencies, scripts, data collection, deployment logic, prompt logic, integration behavior, telemetry, or executable behavior.
4. Keep language calm, enterprise-safe, procurement-safe, CISO-friendly, and legally careful.
5. Confirm that no secrets, credentials, customer references, private prompts, compliance evidence, or private methods are included.

## Issues

Public issues should be limited to non-sensitive repository concerns. Do not open public issues containing vulnerability details, prompt details, deployment logic, customer information, private architecture, compliance evidence, implementation-sensitive details, or commercial negotiation material. Use SECURITY.md for security reporting.

## License and Ownership

The public repository materials are governed by the applicable license files in this repository. Private Enterprise Edition materials, confidential advisory work, customer-specific architectures, and commercial implementation details are not included in the Public Edition.

## Maintainer Discretion

The maintainer may close issues or pull requests that are out of scope, unsafe, duplicative, commercially sensitive, legally sensitive, or inconsistent with the flagship authority role of this repository.
